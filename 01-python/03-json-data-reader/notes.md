## 접근 방식

1. **파일 I/O 및 파싱 구현**: `path.open("r", encoding="utf-8")`와 `json.load()`를 조합하여 외부 JSON 데이터를 파이썬 객체로 역직렬화했습니다.
2. **데이터 유효성 검증**: 데이터 형식이 올바르지 않을 때의 에러 상황을 방지하기 위해 `isinstance`를 사용하여 데이터 구조의 최상단(`list`)과 내부 요소(`dict`)를 순차 검증하고 예외(`ValueError`)를 처리했습니다.
3. **카테고리 개수 집계**: 순서에 상관없이 번갈아가며 들어오는 여러 카테고리 데이터를 한눈에 모아보기 위해 딕셔너리와 루프를 설계하고 누적 카운트 방식을 선택했습니다.

## 새로 배운 내용

### 1. 자원 관리 및 파일 열기 (`with` 문 & `Path.open()`)
* **`with` 문의 역할**: 기존의 `try-finally` 블록 없이도 블록을 벗어나면 파일 객체의 `.close()`가 에러 여부와 관계없이 자동으로 안전하게 호출되는 원리를 배웠습니다.
* **`Path.open()`의 주요 매개변수**:
  * `mode`: 파일을 여는 목적 (기본값 `'r'`은 읽기 전용).
  * `encoding`: 문자 인코딩 규격 (Windows 등에서 한글 깨짐을 방지하기 위해 `'utf-8'` 명시 권장).
  * `errors`: 인코딩 에러 발생 시 처리 정책 (기본값 `'strict'`).
  * `newline`: 줄 바꿈 문자 처리 방식.
  * 반환 타입인 `TextIOWrapper`는 파일 내용을 읽고 쓸 수 있는 파이썬 파일 객체임을 이해했습니다.

### 2. JSON 데이터 읽기 및 파이썬 타입 변환
* **파싱/역직렬화(Deserialization) 메커니즘**: 디스크에 문자열(텍스트) 형태로 저장된 JSON 파일도 `json.load()`를 거치면 파이썬이 즉시 다룰 수 있는 실제 `list`, `dict` 객체 등으로 자동 변환되어 읽힙니다.
* **`json.load()` vs `json.loads()`**:
  * `load(file_object)`: **파일 객체** 자체를 인자로 받아 파일 내부 데이터를 해석합니다.
  * `loads(json_string)`: **JSON 형식의 문자열(String)** 변수를 직접 받아 해석합니다. (`s`는 String을 뜻함)

### 3. 타입 검증 문법 (`isinstance` & `all`)
* **`isinstance(object, classinfo)`**: 특정 객체가 지정된 클래스(자료형)의 인스턴스가 맞는지 확인하는 내장 함수로, 상속 관계까지 지원해 `type() == class`보다 안전하게 쓰입니다.
* **데이터 구조 검증 흐름 (`not isinstance(...) or not all(...)`)**:
  * 1단계: 전체 컨테이너가 `list` 형태인지 검증 (`isinstance(data, list)`)
  * 2단계: 리스트 내부의 개별 항목들이 모두 `dict` 형태인지 검증 (`all(isinstance(item, dict) for item in data)`)
  * 이 두 검증을 통과하지 못하면 올바르지 않은 형식이므로 예외를 발생시킵니다.
* **제너레이터 표현식과 스코프**:
  * `for item in data`에서 `data`는 이전에 `json.load()`를 통해 생성된 **실제 변수**이며, 이를 참조하여 가져옵니다.
  * `item`은 뒤에 적힌 `for item in data` 구문이 먼저 해석되면서 생성되는 **루프용 임시 변수**입니다. (마치 `for i in range`에서 `i`처럼 사용됨)

### 4. 딕셔너리 기반 카테고리 집계 알고리즘
* **변수명 `counts`**: `counts`는 파이썬 내장 함수가 아니라, 우리가 직접 선언한 딕셔너리 변수명(`counts: dict[str, int] = {}`)입니다.
* **`counts.get(category, 0)` 매개변수의 의미**:
  * 첫 번째 인자(`category`): 딕셔너리에서 검색하려는 **키(Key)**이며, 여기서는 개별 레코드의 `"category"` **속성값**(예: `"fruit"`)에 해당합니다.
  * 두 번째 인자(`0`): 해당 키가 딕셔너리에 처음 등장해서 존재하지 않을 때 돌려받을 **기본값**입니다.
* **빈도수 누적(`+ 1`)의 원리**: 데이터가 순서대로 정렬되어 있지 않고 뒤섞여서 번갈아 나오더라도, 매번 꺼낸 카테고리를 기준으로 기존 점수(`get`으로 가져온 값)에 `1`을 더해 다시 딕셔너리에 대입함으로써 카테고리별 누적 수량을 완성합니다.

## 실수와 해결

1. **구문 오류**: `if` 조건문의 끝에 콜론(`:`)을 적는 것을 누락하여 발생했던 `SyntaxError`를 확인하고 조치했습니다.
2. **에러 메시지의 세분화**: 여러 조건을 `or`로 묶을 시 정확히 어떤 세부 조건 때문에 에러가 발생했는지 모호해진다는 점을 깨달았습니다. 더 직관적인 사용자 에러 피드백을 위해 조건을 `if`문 두 개로 분할하여 각각 명확한 에러 메시지를 제공하는 방식이 디버깅과 유지보수 효율 면에서 우수함을 배웠습니다.
